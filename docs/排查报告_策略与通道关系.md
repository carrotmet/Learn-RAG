## 一、用户核心疑问

1. **ChromaDB 中是否也要存各种策略的派生内容？**（摘要、子块、假设问题）
2. **Graph 是否也要从所有策略内容中抽取关键词？**

## 二、设计意图（来自搭建计划文档）

### 2.1 多路召回设计表格

| 召回路 | 技术 | 适用查询特征 | 召回内容 |
|---|---|---|---|
| 语义召回 | **向量检索 + 标准/摘要策略** | 概念理解、语义相似 | 语义相关 chunk |
| 关键词召回 | **FTS5 + 标准策略** | 精准术语、人名、专有名词 | 关键词匹配 chunk |
| 关系召回 | **图检索 + 标准策略** | 实体关联、多跳推理 | 关联实体所在 chunk |
| 摘要召回 | **向量/FTS + 摘要策略** | 高层语义、概述性查询 | 摘要相关 chunk |
| 子块召回 | **向量/FTS + 父子策略** | 细节定位、精确匹配 | 子块命中 → 父块返回 |
| HyDE 召回 | **向量 + 假设性问题策略** | 口语化、词汇鸿沟 | 问题匹配 → 原文返回 |

### 2.2 设计意图解读

- **策略**决定"派生什么内容"（原文/摘要/子块/假设问题）
- **通道**决定"用什么技术索引/检索"
- 不同策略应该对应**不同的通道组合**，而不是全部通道

## 三、当前实现状态

### 3.1 代码结构

```
策略.build(chunks, store, vector=vector, fts=fts, graph=graph)
```

当前测试代码对每个策略都传入了**全部三个通道**。

### 3.2 各通道实际存储内容

#### ChromaDB (向量通道)

| 策略 | 数量 | 内容类型 |
|------|------|---------|
| standard | 10 | 父块原文 ✅ |
| summary | 10 | LLM 生成摘要 |
| parent_child | 55 | 300字子块 |
| hypothetical | 30 | LLM 生成假设问题 |
| **合计** | **105** | |

**问题**: 摘要/子块/问题 都存入了向量库，但向量检索依赖语义相似度：
- 子块（300字）的语义表示可能不完整
- 假设问题是问句形式，与陈述句的查询语义空间不同
- 摘要虽然语义完整，但会引入额外的信息压缩损失

#### FTS5 (全文通道)

| 策略 | 数量 | 内容类型 |
|------|------|---------|
| standard | 10 | 父块原文 ✅ |
| summary | 10 | LLM 生成摘要 |
| parent_child | 55 | 300字子块 |
| hypothetical | 30 | LLM 生成假设问题 |
| **合计** | **105** | |

**问题**: FTS 是关键词精确匹配：
- 子块内容被切分后，关键词被分散到多个子块中
- 假设问题中的疑问词（"什么"、"为什么"）会干扰精确匹配
- 摘要中的元语言（"本文探讨"、"旨在"）不是有效检索词

#### Graph (图谱通道)

| 节点类型 | 数量 |
|---------|------|
| chunk | 10 ✅ |
| keyword | 616 |

**严重问题**: Graph 从**所有策略内容**中抽取了关键词：
- 从子块抽取的片段化关键词: "照片时看到的不仅是你的脸"、"中浓缩出来的"
- 从假设问题抽取的疑问词: "什么是自指"、"为什么被称为"
- 从摘要抽取的元语言: "本文探讨"、"旨在"

这些"污染性"关键词进入了图谱，导致：
1. 图谱节点数膨胀（616 个 keyword，大部分无意义）
2. 图检索时返回不相关结果
3. 违背"图检索 + 标准策略"的设计意图

### 3.3 关键词质量抽样（问题示例）

从 Graph 中抽样的 keyword 节点：
- "不行就写" → 无意义片段
- "但我要先解释" → 口语化过渡
- "中浓缩出来的" → 片段化
- "义层" → 片段化（原词"语义层"）
- "旨在" → 元语言

## 四、问题根因分析

### 4.1 策略 build 方法的设计缺陷

当前策略的 `build` 方法**无条件写入所有传入的通道**：

```python
def build(self, chunks, doc_store, vector=None, fts=None, graph=None):
    # ... 生成派生内容 ...
    if vector: vector.add(items)      # ← 只要有 vector 就写
    if fts: fts.add(items)            # ← 只要有 fts 就写
    if graph: graph.add(items)        # ← 只要有 graph 就写
```

**问题**: 策略本身不决定"应该写入哪些通道"，而是依赖调用方传入的参数。但调用方（测试代码）为了方便传入了全部通道。

### 4.2 策略与通道的映射关系缺失

设计文档中明确了策略与通道的对应关系，但代码中没有体现：

| 策略 | 应该写入的通道 | 当前写入的通道 |
|------|--------------|--------------|
| standard | vector + fts + graph | vector + fts + graph ✅ |
| summary | vector + fts | vector + fts + graph ❌ |
| parent_child | vector + fts | vector + fts + graph ❌ |
| hypothetical | vector | vector + fts + graph ❌ |

### 4.3 GraphChannel 的关键词抽取过于简单

```python
def _extract_keywords(self, text: str, topk: int = 8) -> List[str]:
    words = re.findall(r'[\u4e00-\u9fa5]{2,6}', text)
    # 仅基于词频，无实体识别，无语义过滤
```

问题：
- 没有区分"知识实体"和"功能词/过渡词"
- 子块和假设问题中的内容与标准 chunk 的语义密度不同
- 抽取结果大量噪声

## 五、建议修复方案

### 方案 A: 策略内部决定通道映射（推荐）

在策略类中明确定义支持的通道：

```python
class StandardStrategy(IndexStrategy):
    name = "standard"
    supported_channels = ["vector", "fts", "graph"]  # ← 明确声明
    
class SummaryStrategy(IndexStrategy):
    name = "summary"
    supported_channels = ["vector", "fts"]  # ← graph 不需要
    
class ParentChildStrategy(IndexStrategy):
    name = "parent_child"
    supported_channels = ["vector", "fts"]  # ← graph 不需要
    
class HypotheticalStrategy(IndexStrategy):
    name = "hypothetical"
    supported_channels = ["vector"]  # ← 仅向量语义匹配
```

build 方法改为：
```python
def build(self, chunks, doc_store, channels: Dict[str, Any]):
    # channels = {"vector": v, "fts": f, "graph": g}
    for ch_name, ch in channels.items():
        if ch_name in self.supported_channels and ch is not None:
            ch.add(items)
```

### 方案 B: 调用方显式指定通道（当前架构的轻量修复）

修改测试/调用代码：

```python
# 标准策略 → 全通道
standard.build(chunks, store, vector=vector, fts=fts, graph=graph)

# 摘要策略 → 仅 vector + fts
summary.build(chunks, store, vector=vector, fts=fts, graph=None)

# 父子策略 → 仅 vector + fts
parent_child.build(chunks, store, vector=vector, fts=fts, graph=None)

# 假设问题策略 → 仅 vector
hypothetical.build(chunks, store, vector=vector, fts=None, graph=None)
```

### 方案 C: 优化 GraphChannel 的输入过滤

即使采用方案 B，GraphChannel 也应该过滤输入：

def add(self, items: List[Dict]):

## 八、结论

**当前实现存在"策略与通道关系混乱"的问题**：

1. 所有策略的派生内容都写入了所有通道
2. Graph 通道被严重污染（616 个 keyword 中大部分无意义）
3. 这与设计文档中"不同召回路对应不同策略+通道组合"的意图相违背

**建议立即执行方案 B（轻量修复）**：
- 修改调用代码，为每个策略指定正确的通道组合
- 为 GraphChannel 添加策略过滤
- 重新构建索引

**长期建议实施方案 A**：
- 在策略基类中定义 `supported_channels`
- 由策略自身决定写入哪些通道
- 提高架构清晰度和可维护性
