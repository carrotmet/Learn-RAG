## 三、根因分析

### 3.1 核心问题：`__file__` 路径层级不一致

项目结构中，三个通道类位于不同深度的目录：

```
backend/src/
├── hybrid/
│   ├── document_store.py      ← 第 1 层
│   ├── cli.py
│   └── channels/
│       ├── vector.py          ← 第 2 层（多了一层 channels/）
│       ├── fts.py             ← 第 2 层
│       └── graph.py           ← 第 2 层
```

**问题代码**：三个通道类使用**相同的 `dirname × 3` 逻辑**计算项目根目录：

```python
# document_store.py（正确 ✅）
base = os.path.dirname(os.path.dirname(os.path.dirname(__file__)))
# __file__ = .../backend/src/hybrid/document_store.py
# dirname 1 = .../backend/src/hybrid/
# dirname 2 = .../backend/src/
# dirname 3 = .../backend/          ← 正确指向 backend/
# db_path = backend/data/rag_data.db

# fts.py（错误 ❌）
base = os.path.dirname(os.path.dirname(os.path.dirname(__file__)))
# __file__ = .../backend/src/hybrid/channels/fts.py
# dirname 1 = .../backend/src/hybrid/channels/
# dirname 2 = .../backend/src/hybrid/
# dirname 3 = .../backend/src/      ← 错误！指向了 backend/src/
# db_path = backend/src/data/rag_data.db  ← 幽灵数据库
```

### 3.2 路径漂移的完整映射

| 类 | 文件路径 | `dirname×3` 结果 | 默认数据路径 |
|---|---|---|---|
| DocumentStore | `hybrid/document_store.py` | `backend/` | `backend/data/rag_data.db` ✅ |
| FTSChannel | `hybrid/channels/fts.py` | `backend/src/` | `backend/src/data/rag_data.db` ❌ |
| VectorChannel | `hybrid/channels/vector.py` | `backend/src/` | `backend/src/chroma_db_hybrid/` ❌ |
| GraphChannel | `hybrid/channels/graph.py` | `backend/src/` | `backend/src/data/knowledge_graph.pkl` ❌ |

### 3.3 为什么"有时正常，有时分裂"

**正常路径**（数据统一到 `backend/data/rag_data.db`）：
- `cli.py` 显式传递 `db_path` 给所有组件
- 测试脚本显式定义 `DB_PATH = .../backend/data/rag_data.db`

**分裂路径**（数据分散到 `backend/src/` 下）：
- 直接实例化 `FTSChannel()`（不传 db_path）
- 直接实例化 `VectorChannel()`（不传 persist_dir）
- 直接实例化 `GraphChannel()`（不传 path）
- 此时各组件使用各自的默认路径，导致数据写到了不同位置

**具体触发场景**：
1. 测试脚本 `test_phase1.py` 中 `fts_ch = FTSChannel()` → 不传 db_path → 创建 `backend/src/data/rag_data.db`
2. `MultiRecall.__init__` 中 `self.fts = fts or FTSChannel(db_path=doc_store.db_path)` → 正常（因为传了 db_path）
3. 但 `MultiRecall.__init__` 中 `self.vector = vector or VectorChannel()` → **不传 persist_dir** → 创建 `backend/src/chroma_db_hybrid/`

### 3.4 数据分裂的实际影响

```
用户执行 CLI 索引：
  → DocumentStore → backend/data/rag_data.db ✅
  → FTSChannel(db_path=...) → backend/data/rag_data.db ✅
  → VectorChannel(persist_dir=...) → backend/src/chroma_db_hybrid/ ❌
  → GraphChannel → backend/src/data/knowledge_graph.pkl ❌

用户执行检索：
  → MultiRecall() → VectorChannel() → backend/src/chroma_db_hybrid/（空或旧数据）
                  → FTSChannel(db_path=...) → backend/data/rag_data.db（正确）
  → 结果：vector 召回为空或错误，fts 召回正常
```

## 四、问题定级

| 维度 | 评估 |
|------|------|
| 严重性 | 🔴 **高** — 数据分裂导致检索结果不完整或错误 |
| 影响范围 | 所有使用默认参数初始化通道的代码路径 |
| 数据丢失风险 | 中 — 旧数据仍在，但新数据可能写入错误位置 |
| 修复复杂度 | 低 — 只需统一路径计算逻辑 |

## 五、修复建议（未执行，待用户确认）

### 方案 A：统一使用项目根目录常量（推荐）

在 `hybrid/__init__.py` 或独立模块中定义 `PROJECT_ROOT`，所有组件引用该常量：

```python
# hybrid/config.py
import os
PROJECT_ROOT = os.path.dirname(os.path.dirname(os.path.dirname(os.path.abspath(__file__))))
DEFAULT_DB_PATH = os.path.join(PROJECT_ROOT, "backend", "data", "rag_data.db")
DEFAULT_CHROMA_DIR = os.path.join(PROJECT_ROOT, "backend", "chroma_db_hybrid")
```

### 方案 B：修正 dirname 层级

`channels/` 下的类需要多调用一次 `dirname`：

```python
# fts.py / vector.py / graph.py
base = os.path.dirname(os.path.dirname(os.path.dirname(os.path.dirname(__file__))))
# 4 层 dirname → 指向 backend/
```

### 方案 C：废弃默认参数，强制显式传参

所有组件初始化时必须显式传入 `db_path` / `persist_dir`，不传则抛异常。从代码层面杜绝默认路径不一致的问题。
